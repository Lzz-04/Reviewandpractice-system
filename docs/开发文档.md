## 5. 后端开发规范

### 5.1 分层规范

后端采用标准三层结构：

```text
Controller -> Service -> Mapper -> Database
```

各层职责：

| 层 | 职责 |
|---|---|
| Controller | 接收请求、参数绑定、返回统一响应 |
| Service | 编写业务逻辑、事务控制、权限和数据校验 |
| Mapper | 数据库访问，继承 MyBatis-Plus `BaseMapper` |
| Entity | 与数据库表字段对应 |
| DTO | 接口请求或响应对象 |

### 5.2 Controller 开发规范

Controller 只做轻量处理：

- 定义接口路径和 HTTP 方法
- 接收请求参数
- 调用 Service
- 使用 `ApiResponse` 返回结果

示例：

```java
@RestController
@RequestMapping("/api/subjects")
@RequiredArgsConstructor
public class SubjectController {

    private final SubjectService subjectService;

    @GetMapping
    public ApiResponse<List<Subject>> getAll() {
        return ApiResponse.ok(subjectService.getAll(SecurityUtil.getCurrentUserId()));
    }
}
```

### 5.3 Service 开发规范

Service 负责核心业务逻辑：

- 参数合法性校验
- 数据归属校验
- 业务规则判断
- 调用 Mapper
- 事务控制

需要事务的方法使用：

```java
@Transactional
```

典型场景：

- 批量删除
- 交卷并写入多张表
- 删除用户及其关联数据
- 导入题目

### 5.4 Mapper 开发规范

常规 CRUD 使用 MyBatis-Plus：

```java
public interface SubjectMapper extends BaseMapper<Subject> {
}
```

复杂查询优先使用 `LambdaQueryWrapper`：

```java
new LambdaQueryWrapper<Subject>()
    .eq(!SecurityUtil.isAdmin(), Subject::getUserId, userId)
    .orderByAsc(Subject::getSortOrder)
```

### 5.5 Entity 开发规范

实体类使用 Lombok 简化代码：

```java
@Data
@TableName("subjects")
public class Subject {
    @TableId(type = IdType.AUTO)
    private Integer id;
    private String name;
    private Long userId;
}
```

字段命名采用 Java 驼峰，数据库字段采用下划线，MyBatis-Plus 已启用自动映射：

```yaml
mybatis-plus:
  configuration:
    map-underscore-to-camel-case: true
```

### 5.6 异常处理规范

业务异常统一抛出：

```java
throw new BusinessException("科目名称不能为空");
```

需要指定状态码时：

```java
throw new BusinessException(403, "无权访问");
```

全局异常处理类：

```text
server/src/main/java/com/examreview/exception/GlobalExceptionHandler.java
```

### 5.7 统一响应规范

所有接口统一返回：

```java
ApiResponse<T>
```

成功示例：

```java
return ApiResponse.ok(data, "创建成功");
```

失败由异常处理器统一转换。

响应格式：

```json
{
  "code": 200,
  "data": {},
  "message": "success"
}
```

### 6.3 状态管理规范

跨页面共享状态使用 Pinia，文件放在：


现有 Store：

| Store | 作用 |
|---|---|
| `auth.js` | 登录态、Token、用户信息 |

### 6.4 API 调用规范

前端统一通过 `useApi()` 调用后端接口。

文件：

```text
client/composables/useApi.js
```

调用示例：

```js
const api = useApi()
const subjects = await api.get('/subjects')
const created = await api.post('/subjects', form)
await api.put(`/subjects/${id}`, form)
await api.delete(`/subjects/${id}`)
```

不要在页面中直接使用 `$fetch` 调业务接口，否则会绕过 JWT 注入和 401 统一处理。

### 6.5 路由鉴权规范

全局路由守卫文件：

```text

```

当前白名单：

```js
const publicRoutes = ['/login', '/register']
```

## 7. 认证与权限开发说明

### 7.1 登录态流转

```text
登录成功
  -> 后端返回 token
  -> auth.js 保存 token 和 user
  -> localStorage 保存 auth_token 和 auth_user
  -> useApi 后续请求自动带 Authorization
```

### 7.2 后端获取当前用户

在业务代码中获取当前用户 ID：

```java
Long userId = SecurityUtil.getCurrentUserId();
```

判断管理员：

```java
if (!SecurityUtil.isAdmin()) {
    throw new BusinessException(403, "无权访问");
}
```

### 7.3 数据隔离写法

普通用户查询自己的数据，管理员不过滤：

```java
new LambdaQueryWrapper<Subject>()
    .eq(!SecurityUtil.isAdmin(), Subject::getUserId, userId)
```

新增业务表如果属于用户私有数据，应增加：

```text
user_id BIGINT NOT NULL DEFAULT 0
```

并在 Service 层所有查询、更新、删除中做用户隔离。

## 9. 新增模块开发流程

新增一个标准业务模块时，推荐按以下步骤开发。

### 9.1 后端开发步骤

1. 设计数据库表，在 `schema.sql` 中新增建表或增量字段。
2. 新建 Entity，放在 `entity` 包下。
3. 新建 Mapper，继承 `BaseMapper<Entity>`。
4. 新建 DTO，用于请求或响应。
5. 新建 Service 接口。
6. 新建 ServiceImpl，实现业务逻辑。
7. 新建 Controller，对外提供 REST API。
8. 在需要事务的方法上添加 `@Transactional`。
9. 通过 `SecurityUtil` 做用户隔离和权限校验。
10. 启动后端，使用浏览器或接口工具测试。

### 9.2 前端开发步骤

1. 在 `pages` 下新建页面。
2. 如果需要复用 UI，先抽取到 `components`。
3. 如果需要跨页面状态，新建或扩展 Pinia Store。
4. 使用 `useApi()` 调后端接口。
5. 增加加载状态、空状态和错误提示。
6. 如果页面需要登录，默认会被全局路由守卫保护。
7. 如果是匿名页面，加入 `auth.global.js` 白名单。

### 9.3 模块开发模板

后端 Controller 模板：

```java
@RestController
@RequestMapping("/api/example")
@RequiredArgsConstructor
public class ExampleController {

    private final ExampleService exampleService;

    @GetMapping
    public ApiResponse<List<Example>> list() {
        return ApiResponse.ok(exampleService.list(SecurityUtil.getCurrentUserId()));
    }

    @PostMapping
    public ApiResponse<Example> create(@RequestBody Example example) {
        return ApiResponse.ok(exampleService.create(example, SecurityUtil.getCurrentUserId()), "创建成功");
    }
}
```

前端调用模板：

```js
const api = useApi()
const list = ref([])
const loading = ref(false)

async function loadData() {
  loading.value = true
  try {
    list.value = await api.get('/example')
  } finally {
    loading.value = false
  }
}
```

### 10.3 常见接口错误

| 错误 | 可能原因 | 处理 |
|---|---|---|
| 401 请先登录 | 没带 Token 或 Token 过期 | 重新登录 |

## 12. 常见问题

### 12.1 后端启动失败，提示数据库连接失败

检查：

- MySQL 是否启动
- 数据库账号密码是否正确
- `DB_PASSWORD` 环境变量是否设置
- `application.yml` 是否存在
- JDBC URL 是否正确

### 12.2 JWT 报错或登录后接口 401

检查：

- `JWT_SECRET` 是否配置
- JWT 密钥长度是否足够
- 前端 `localStorage` 中是否存在 `auth_token`
- 请求头是否包含 `Authorization: Bearer <token>`

### 12.3 前端访问接口跨域

检查：

- 后端是否运行在 `8080`
- 前端是否运行在 `3000` 或 `3001`
- `SecurityConfig` 中 CORS 是否包含当前前端地址
- `nuxt.config.ts` 中 `apiBase` 是否指向 `http://localhost:8080/api`

### 12.4 数据重复初始化

当前配置：

```yaml
spring.sql.init.mode: always
spring.sql.init.continue-on-error: true
```

系统每次启动都会尝试执行 `schema.sql` 和 `data.sql`。如果种子数据重复，需要检查 `data.sql` 是否使用了防重复写法。

### 12.5 中文乱码

检查：

- 数据库字符集是否为 `utf8mb4`
- JDBC URL 是否包含 `characterEncoding=UTF-8`
- 请求体是否使用 UTF-8 编码
- 文件导入时源文件编码是否正确

## 13. 开发注意事项

- 不要绕过 `useApi()` 直接请求业务接口。
- 后端新增接口默认需要登录，除非在 `SecurityConfig` 中显式放行。
- 新增用户私有数据表时必须加 `user_id`。
- 普通用户数据查询必须按 `user_id` 过滤。
- 删除操作要先检查关联数据，必要时使用事务。
- 批量写入、批量删除、交卷、用户级联删除必须加事务。
- 前端页面需要处理 loading、空数据和错误提示。
- 后端业务错误使用 `BusinessException`，不要直接返回字符串。
- 接口返回统一使用 `ApiResponse`。
- 修改数据库字段后，同步更新实体类和前端字段。
