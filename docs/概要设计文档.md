# 期末复习刷题系统概要设计文档

## 1. 文档概述

### 1.1 文档目的

本文档用于说明期末复习刷题系统的总体设计方案，包括系统架构、模块划分、数据结构、接口设计、安全设计、异常处理和部署运行方案，为后续详细设计、编码实现、测试验收和项目答辩提供依据。

### 1.2 项目简介

期末复习刷题系统是一个面向大学生的 Web 刷题复习平台，支持用户登录注册、科目管理、章节管理、题库管理、章节练习、模拟考试、错题本、学习统计、文件导入和 AI 辅助出题等功能。系统通过结构化题库、即时判分、错题追踪和学习数据分析，帮助用户形成完整的复习闭环。

### 1.3 设计范围

本文档覆盖以下内容：

- 系统总体架构设计
- 前后端技术架构设计
- 后端分层结构设计
- 前端页面与状态管理设计
- 功能模块概要设计
- 数据库概要设计
- 主要接口概要设计
- 认证授权与数据隔离设计
- 异常处理与部署运行设计

## 2. 系统目标

### 2.1 业务目标

| 目标 | 说明 |
|---|---|
| 复习内容结构化 | 用户可按科目、章节组织题库 |

| 目标 | 说明 |

## 3. 总体架构设计

### 3.1 系统架构

系统采用前后端分离架构。前端负责页面展示、用户交互、状态管理和接口调用；后端负责业务处理、权限校验、数据持久化、文件解析和 AI 服务调用；数据库负责存储用户、题库、考试、答题和错题等核心数据。

```text
用户浏览器
  |
  | HTTP / JSON
  v
Nuxt 3 前端应用
  |
  | REST API + JWT
  v
Spring Boot 后端服务
  |
  | MyBatis-Plus
  v
MySQL 数据库

Spring Boot 后端服务
  |
  | WebClient
  v
DeepSeek API
```

### 3.2 技术架构

| 层级 | 技术 |
|---|---|
| 前端框架 | Nuxt 3、Vue 3 Composition API |

## 4. 功能模块概要设计

### 4.1 认证授权模块

认证授权模块负责用户注册、登录、获取当前用户信息、修改密码和接口鉴权。

主要功能：

- 用户注册：校验用户名和密码，使用 BCrypt 加密密码后写入数据库。
- 用户登录：通过 Spring Security `AuthenticationManager` 校验账号密码。
- Token 生成：登录或注册成功后生成 JWT。
- 接口鉴权：业务接口必须携带 `Authorization: Bearer <token>`。
- 前端守卫：未登录用户访问业务页面时跳转登录页。

核心文件：

| 类型 | 文件 |
|---|---|
| 后端接口 | `AuthController.java` |
| 后端业务 | `AuthServiceImpl.java` |
| JWT 工具 | `JwtUtil.java` |
| JWT 过滤器 | `JwtAuthenticationFilter.java` |
| 用户主体 | `UserPrincipal.java` |
| 前端状态 | `stores/auth.js` |
| 请求封装 | `composables/useApi.js` |
| 路由守卫 | `middleware/auth.global.js` |

认证流程：

```text
用户提交登录表单
  -> 前端 authStore.login()
  -> useApi.post('/auth/login')
  -> AuthController.login()
  -> AuthServiceImpl.login()
  -> AuthenticationManager 校验用户名密码
  -> JwtUtil.generateToken()
  -> 返回 token 和用户信息
  -> 前端保存 auth_token 和 auth_user
```

- 删除科目前检查是否存在关联章节

核心文件：

| 类型 | 文件 |
|---|---|
| 后端接口 | `SubjectController.java` |
| 后端业务 | `SubjectServiceImpl.java` |

## 5. 数据库概要设计

### 5.1 数据库设计原则

- 使用 MySQL 8.x 作为关系型数据库。
- 主键使用自增 ID。
- 业务表通过 `user_id` 实现用户数据隔离。
- 题目选项使用 JSON 字段存储，兼容不同题型。
- 常用查询字段建立索引，提高查询效率。
- 错题表通过唯一索引防止重复记录。

### 5.2 核心表

| 表名 | 说明 | 主要字段 |
|---|---|---|
| `users` | 用户表 | `id`、`username`、`password`、`nickname`、`role` |
| `subjects` | 科目表 | `id`、`name`、`description`、`sort_order`、`user_id` |
| `chapters` | 章节表 | `id`、`subject_id`、`name`、`sort_order`、`user_id` |
| `questions` | 题目表 | `id`、`chapter_id`、`subject_id`、`type`、`content`、`options`、`answer`、`difficulty`、`user_id` |
| `exam_papers` | 试卷表 | `id`、`subject_id`、`title`、`duration`、`question_count`、`user_id` |
| `exam_questions` | 试卷题目关联表 | `id`、`exam_id`、`question_id`、`sort_order` |
| `exam_records` | 考试记录表 | `id`、`exam_id`、`subject_id`、`score`、`status`、`session_id`、`user_id` |
| `answer_records` | 答题记录表 | `id`、`question_id`、`exam_id`、`session_id`、`selected_answer`、`is_correct`、`user_id` |
| `wrong_questions` | 错题表 | `id`、`question_id`、`subject_id`、`chapter_id`、`wrong_count`、`mastered`、`user_id` |

### 5.3 表关系

```text
users 1 ── N subjects
subjects 1 ── N chapters
chapters 1 ── N questions
subjects 1 ── N questions

subjects 1 ── N exam_papers
exam_papers 1 ── N exam_questions
questions 1 ── N exam_questions

users 1 ── N exam_records
users 1 ── N answer_records
users 1 ── N wrong_questions

questions 1 ── N answer_records
questions 1 ── N wrong_questions
```

### 5.4 关键约束

| 约束 | 说明 |
|---|---|
| `users.username UNIQUE` | 用户名唯一 |
| `questions.type CHECK` | 题型只能是 `single`、`multiple`、`judge` |
| `questions.difficulty CHECK` | 难度范围 1-5 |
| `wrong_questions uk_user_question` | 同一用户同一题只保留一条错题 |

### 5.5 主要索引

| 索引 | 作用 |
|---|---|
| `idx_questions_chapter` | 按章节查询题目 |
| `idx_questions_subject` | 按科目查询题目 |
| `idx_questions_type` | 按题型筛选 |
| `idx_questions_difficulty` | 按难度筛选 |
| `idx_answer_records_session` | 按答题批次查询 |
| `idx_exam_records_user` | 查询用户考试记录 |
| `idx_wrong_questions_user` | 查询用户错题 |
| `idx_chapters_subject` | 查询科目下章节 |

## 6. 接口概要设计

### 6.1 接口设计规范

后端接口采用 REST 风格，统一以 `/api` 开头。请求和响应数据采用 JSON 格式。除登录、注册外，其他接口均需要 JWT 认证。

统一响应结构：

```json
{
  "code": 200,
  "data": {},
  "message": "success"
}
```

错误响应示例：

```json
{
  "code": 400,
  "data": null,
  "message": "科目名称不能为空"
}
```

### 6.2 主要接口

#### 认证接口

| 方法 | 路径 | 说明 | 认证 |
|---|---|---|---|
| POST | `/api/auth/register` | 用户注册 | 否 |
| POST | `/api/auth/login` | 用户登录 | 否 |
| GET | `/api/auth/me` | 获取当前用户 | 是 |
| PUT | `/api/auth/password` | 修改密码 | 是 |

#### 科目接口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/subjects` | 获取科目列表 |
| GET | `/api/subjects/{id}` | 获取科目详情 |
| POST | `/api/subjects` | 创建科目 |
| PUT | `/api/subjects/{id}` | 更新科目 |
| DELETE | `/api/subjects/{id}` | 删除科目 |

#### 章节接口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/subjects/{subjectId}/chapters` | 获取科目下章节 |

#### 题库接口

| 方法 | 路径 | 说明 |

| GET | `/api/questions` | 分页查询题目 |

## 7. 安全设计

### 7.1 认证机制

系统使用 JWT 作为前后端分离场景下的认证凭证。用户登录成功后，后端签发 JWT，前端保存到 `localStorage`。后续请求通过请求头携带 Token：

```text
Authorization: Bearer <token>
```

后端通过 `JwtAuthenticationFilter` 拦截请求，校验 Token 后将用户信息写入 Spring Security 上下文。

### 7.2 密码安全

用户密码不以明文形式存储。注册和修改密码时，后端使用 `BCryptPasswordEncoder` 对密码进行加密。

### 7.3 权限控制

权限分为普通用户和管理员：

| 角色 | 权限 |
|---|---|
| `user` | 管理自己的科目、章节、题目、练习、考试、错题和统计 |
| `admin` | 查看所有用户数据，删除普通用户及其关联数据 |

管理员判断通过 `SecurityUtil.isAdmin()` 完成。

### 7.4 数据隔离

主要业务表包含 `user_id` 字段。普通用户查询和操作数据时，服务层自动添加用户过滤条件：

```java
.eq(!SecurityUtil.isAdmin(), Entity::getUserId, userId)
```

该设计保证普通用户只能访问自己的数据，管理员可以在后台跨用户管理。

### 7.5 跨域设计

后端通过 Spring Security CORS 配置允许前端开发端口访问接口：

- `http://localhost:3000`
- `http://localhost:3001`

允许的请求方法：

- GET
- POST
- PUT
- DELETE
- OPTIONS

## 8. 异常处理设计

系统使用统一异常处理类 `GlobalExceptionHandler`，将异常转换为统一响应结构。

| 异常类型 | 处理方式 |
|---|---|
| `BusinessException` | 返回业务错误码和错误消息 |
| 参数校验异常 | 汇总字段错误信息并返回 400 |
| 登录认证异常 | 返回 401 |
| JSON 解析异常 | 返回请求数据格式错误 |
| 未捕获异常 | 记录日志，返回 500 |

统一错误格式：

```json
{
  "code": 400,
  "data": null,
  "message": "错误说明"
}
```

## 10. 关键业务流程设计

### 10.1 登录鉴权流程

```text
用户输入账号密码
  -> 前端提交登录请求
  -> 后端校验账号密码
  -> 生成 JWT
  -> 返回用户信息和 Token
  -> 前端保存 Token
  -> 后续请求自动携带 Token
```

### 10.2 章节练习流程

```text
用户选择科目和章节
  -> 选择练习模式
  -> 系统返回题目列表
  -> 用户逐题作答
  -> 后端判定答案
  -> 保存答题记录
  -> 答错则加入错题本
  -> 前端显示结果和解析
```

### 10.3 模拟考试流程

```text
用户选择科目、题量、考试时长
  -> 系统自动组卷
  -> 用户开始考试
  -> 前端倒计时
  -> 用户答题
  -> 可暂停或恢复
  -> 交卷
  -> 后端统一判分
  -> 保存考试记录和答题记录
  -> 展示考试结果
```

### 10.4 错题收集流程

```text
练习或考试答错
  -> 后端调用错题服务
  -> 查询 wrong_questions 是否已有记录
  -> 已存在则 wrong_count + 1
  -> 不存在则新增错题记录
  -> 用户可在错题本中复习和标记掌握
```

### 10.5 管理员删除用户流程

```text
管理员进入后台
  -> 查询用户列表
  -> 选择删除普通用户
  -> 校验管理员权限
  -> 校验不能删除自己
  -> 校验不能删除管理员
  -> 删除答题记录、错题、考试记录
  -> 删除试卷和试卷题目关联
  -> 删除题目、章节、科目
  -> 删除用户账号
```

## 12. 非功能设计

### 12.1 可维护性

- 后端采用 Controller、Service、Mapper 分层结构。
- DTO 与 Entity 分离，降低接口数据和数据库结构耦合。
- 统一响应和统一异常处理，减少重复代码。
- 前端使用 Pinia 管理跨页面状态。

### 12.2 可扩展性

- 新增业务模块时可按现有 Controller、Service、Mapper 模式扩展。
- 题型通过 `type` 字段区分，后续可扩展填空题、简答题等题型。
- AI 服务独立封装，后续可替换模型供应商。
- 文件导入解析逻辑集中在导入服务中，后续可扩展更多格式。

### 12.3 安全性

- 登录密码使用 BCrypt 加密。
- JWT 具有过期时间。
- 后端接口统一鉴权。
- 普通用户数据通过 `user_id` 隔离。
- 管理员接口进行角色校验。

### 12.4 性能设计

- 对题目、答题记录、考试记录、错题记录等高频查询字段建立索引。
- 题库列表使用分页查询。
- 统计数据按用户维度过滤，减少无关数据扫描。
- 前端通过组件化和状态管理减少重复请求和重复状态维护。

## 13. 概要设计总结

期末复习刷题系统采用前后端分离架构，后端以 Spring Boot 为核心，使用 Spring Security + JWT 实现统一认证授权，使用 MyBatis-Plus 操作 MySQL 数据库；前端采用 Nuxt 3 + Vue 3 + Element Plus 实现页面交互和状态管理。

系统围绕“科目 -> 章节 -> 题目 -> 练习/考试 -> 错题/统计”的学习闭环进行设计，模块边界清晰，数据结构稳定，具备较好的可维护性和扩展性。基础架构模块提供登录鉴权、用户隔离和后台管理能力，题库、练习、考试、错题和统计模块共同构成核心业务流程，AI 和文件导入模块作为增强能力提升系统实用性。
